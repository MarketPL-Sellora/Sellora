# Data Flow & Interaction Specification

## 1. Високорівнева схема (High-Level Architecture)

Система базується на сучасній REST API архітектурі з чітким розділенням клієнтської та серверної
частин:

* **Frontend Layer (Client):** Побудований на **Vue 3** (Composition API) + **Vite**. Відповідає за
  UI, валідацію форм на боці клієнта, керування станом (Pinia) та зберігання JWT-токена.
* **Backend Layer (API):** Побудований на **Java Spring Boot**. Виконує роль REST API, обробляє
  бізнес-логіку, валідацію транзакцій та авторизацію.
* **Data Layer (Storage):** Реляційна база даних **PostgreSQL**, яка забезпечує надійне збереження
  даних про користувачів, товари, замовлення та групи.

## 2. Сценарії взаємодії (Data Flow)

Шлях даних (Request/Response) для 3-х ключових функцій маркетплейсу:

### Сценарій А: Авторизація користувача (Login)

1. **User:** Вводить Email та Пароль, натискає "Увійти".
2. **Frontend:** Формує DTO і відправляє `POST` запит на `/api/auth/login`.

* *Payload:* `{"email": "user@mail.com", "password": "123"}`

3. **Backend:** Звіряє хеш пароля в Postgres. Генерує JWT-токен.
4. **Backend:** Повертає успішну відповідь `200 OK` разом із токеном.

* *Response:* `{"accessToken": "eyJhb...", "user": {"id": 1, "name": "Vadym"}}`

5. **Frontend:** Зберігає токен у Pinia (або localStorage) та перенаправляє користувача в Кабінет.

### Сценарій Б: Оформлення замовлення (Checkout)

1. **User:** Заповнює дані доставки і тисне "Підтвердити замовлення".
2. **Frontend:** Відправляє `POST` запит на `/api/orders`, прикріпивши JWT-токен у заголовок
   `Authorization`.

* *Payload:*
  `{"items": [{"productId": 45, "qty": 1}], "delivery": {"type": "NOVA_POSHTA", "city": "Київ"}, "paymentMethod": "LIQPAY"}`

3. **Backend:** Валідує наявність товару, створює запис замовлення в БД зі статусом "Очікує оплати".
4. **Backend:** Повертає `201 Created` з деталями для оплати.

* *Response:* `{"orderId": 801, "totalPrice": 45699, "paymentUrl": "https://liqpay.ua/..."}`

5. **Frontend:** Редіректить користувача на платіжний шлюз.

### Сценарій В: Ініціація "Групової покупки" (Group Buy)

1. **User:** Тисне акцентну кнопку "Почати групову покупку" в картці товару.
2. **Frontend:** Відправляє `POST` запит на `/api/group-buys` з JWT-токеном ініціатора.

* *Payload:* `{"productId": 89}`

3. **Backend:** Створює нову сесію збору в БД, генерує унікальне посилання та встановлює таймер (
   наприклад, 24 години).
4. **Backend:** Повертає `201 Created` з даними збору.

* *Response:*
  `{"groupId": "gb-992", "expiresAt": "2026-04-09T14:00:00Z", "shareableLink": "sellora.com/group/gb-992", "requiredParticipants": 3}`

5. **Frontend:** Відкриває модальне вікно (Overlay) з таймером та згенерованим посиланням для
   друзів.

## 3. Механізм передачі даних та Statelessness

* **Використання DTO (Data Transfer Objects):** Обмін даними між Vue.js та Java відбувається
  виключно у форматі JSON через узгоджені DTO. Ми не передаємо "сирі" сутності бази даних на клієнт,
  а лише ті поля, які необхідні для відображення конкретного UI.
* **Принцип Statelessness:** Сервер Java не зберігає стан сесії користувача у своїй пам'яті. Кожен
  HTTP-запит від клієнта містить JWT-токен, який бекенд розшифровує для ідентифікації. Це дозволяє
  зробити API незалежним та готовим до легкого масштабування.
