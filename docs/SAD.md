# SAD-001: Internal View

**Статус:** Accepted  
**Дата:** 07.04.2026  
**Проєкт:** Sellora  

## 1. Аналіз логіки та структури (Input)
* **Складність бізнес-правил:** Висока. Проєкт включає не лише базові CRUD-операції, а й складну логіку фінансових транзакцій (утримання/списання/повернення коштів), управління таймерами механіки «Group Buy», захист від стану гонки (Race conditions) при резервуванні місць у групі та жорсткий контроль товарних залишків. Це вимагає повної ізоляції критичної логіки від коду фреймворку.
* **Джерела даних:** Основне сховище — реляційна БД PostgreSQL 16 для забезпечення суворих ACID-транзакцій під час фінансових операцій та управління замовленнями.
* **Зовнішні залежності:** Інтеграція з платіжним еквайрингом (MonoPay API), хмарним сховищем для оптимізації медіа (Cloudinary Free Tier) та сервісом генерації JWT-токенів для авторизації.

## 2. Вибір патерну
**Обраний патерн:** Clean Architecture (або Onion Architecture).

**Чому:** З огляду на стратегічний вибір на користь гнучкості, Clean Architecture дозволить зробити бізнес-логіку («серце» маркетплейсу Sellora) незалежною від фреймворку (Spring Boot), бази даних та зовнішніх API. Це дасть змогу писати швидкі Unit-тести для критичних правил (наприклад, перевірка лімітів групи, валідація цін, зміна статусів) без підключення до реальної бази чи платіжного шлюзу. У майбутньому, при зміні або розширенні зовнішніх сервісів, ядро системи залишиться недоторканим.

## 3. Визначення відповідальності шарів (Layers)

| Шар | Назва (в проєкті) | Відповідальність та компоненти |
| :--- | :--- | :--- |
| **Presentation Layer** | API / Web | Приймає HTTP-запити. Містить REST Controllers, DTOs (Data Transfer Objects), конфігурацію OpenAPI/Swagger та Middlewares (ControllerAdvice) для глобального перехоплення помилок. Не містить жодної бізнес-логіки обчислень. |
| **Application Layer** | Business Logic / Use Cases / Services | Реалізує конкретні сценарії застосунку (Use Cases): «Створити сесію Group Buy», «Обробити вебхук від LiqPay», «Сформувати замовлення». Оркеструє роботу, звертаючись до Domain Layer, і використовує абстрактні інтерфейси репозиторіїв. |
| **Domain Layer** | Core / Entities | Найвищий рівень абстракції, "чиста Java" (без анотацій Spring чи Hibernate). Містить ключові сутності (`User`, `Product`, `GroupBuySession`, `Order`) та внутрішні бізнес-правила (наприклад, метод `canJoinGroup()`, перевірка статусу сесії). |
| **Infrastructure Layer** | Persistence / External | Фізична реалізація зовнішніх інтерфейсів. Реалізація доступу до PostgreSQL через Spring Data JPA (з мапінгом чистого Domain у БД-сутності). Реалізація HTTP-клієнтів для відправки запитів у LiqPay та завантаження фотографій у Cloudinary. Реалізація Spring Security (JWT). |
