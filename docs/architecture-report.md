# РОЗДІЛ: АРХІТЕКТУРА ФРОНТЕНД-ДОДАТКУ ТА КОМПОНЕНТНА СТРУКТУРА

## 1. Загальна архітектура системи

Фронтенд-частина платформи розроблена як SPA (Single Page Application — Односторінковий вебдодаток). Такий підхід забезпечує високу швидкість взаємодії з користувачем, оскільки завантаження сторінки відбувається один раз, а подальша навігація та оновлення даних здійснюються динамічно через API-запити без перезавантаження браузера.

В основі архітектури лежить модульно-компонентний підхід із чітким розділенням зон відповідальності (Separation of Concerns). Система поділена на логічні шари:

**UI Шари (Views, Components, Layouts):** Відповідають виключно за рендеринг інтерфейсу та обробку подій користувача.

**Шар стану (State Management / Pinia):** Акумулює бізнес-логіку додатку, зберігає глобальні дані та керує їх оновленням.

**Шар мережевої взаємодії (API / Axios):** Забезпечує комунікацію з бекенд-сервером.

## 2. Технологічний стек

Для реалізації клієнтської частини було обрано такий стек технологій:

**Vue 3 (Composition API):** Сучасний реактивний фреймворк. Використання Composition API дозволило гнучко групувати логіку за функціональними ознаками (хуки, обчислювальні властивості) та забезпечило високу перевикористовуваність коду.

**Vite:** Швидкий інструмент збірки проєкту, що забезпечує миттєве гаряче оновлення модулів (HMR) під час розробки та оптимізований бандлінг для продакшену.

**Tailwind CSS:** Утилітарний CSS-фреймворк, який використовується для стилізації. Він дозволив реалізувати адаптивний та адаптований під мобільні пристрої (Mobile-First) інтерфейс без написання громіздких медіа-запитів вручну.

## 3. Структура каталогу проєкту

Проєкт має чітку та масштабовану структуру папок у каталозі `src/`, що полегшує підтримку та розширення коду:

```
src/
 ├── api/          # Налаштування HTTP-клієнта (Axios) та перехоплювачів
 ├── assets/       # Статичні ресурси (зображення, глобальні стилі CSS)
 ├── components/   # Перевикористовувані UI-компоненти, розділені за модулями:
 │    ├── cabinet/ # Компоненти особистого кабінету продавця
 │    ├── forms/   # Компоненти форм (створення товару, магазину тощо)
 │    ├── layout/  # Глобальна обгортка (Header, Footer, Sidebar, Breadcrumbs)
 │    ├── modals/  # Модальні вікна (авторизація, кошик, групові покупки)
 │    └── product/ # Картки товарів, галереї, таби
 ├── constants/    # Статичні словники, константи та контент сторінок
 ├── hooks/        # Користувацькі композиційні функції (Composables)
 ├── router/       # Конфігурація маршрутизації (Vue Router)
 ├── state/        # Централізовані сховища даних (Pinia Stores)
 └── views/        # Головні сторінки-екрани додатку (Home, Product, Order)
```

## 4. Централізоване управління станом (State Management)

Для менеджменту глобального стану додатку інтегровано сховище Pinia, яке є офіційним та найбільш оптимізованим рішенням для Vue 3. Бізнес-логіка рознесена по окремих ізольованих сторах:

**`userStore`:** Керує станом автентифікації користувача (сесії, токени), профілем покупця та даними особистого кабінету продавця (`sellerStore`). Обробляє запити на створення/оновлення магазинів.

**`productStore`:** Відповідає за каталог товарів — завантаження списку продуктів із фільтрацією, отримання детальної інформації про конкретний товар за ID, а також завантаження зображень продуктів на сервер.

**`cartStore`:** Локальне сховище кошика користувача. Обчислює загальну кількість обраних товарів, групує їх за продавцями та вираховує фінальну вартість замовлення.

**`groupBuyStore`:** Логічне ядро унікальної фічі маркетплейсу. Керує активними зборами для групових покупок користувача, завантажує сесії розпродажів та оновлює їх статус.

**`categoryStore`:** Зберігає дерево категорій товарів для навігаційного меню та фільтрації.

## 5. Мережевий шар та взаємодія з REST API

Взаємодія з бекендом реалізована за допомогою бібліотеки Axios. У модулі `api/axios.ts` налаштовано глобальний екземпляр `apiClient`:

**Базовий URL:** Усі запити автоматично направляються на єдину точку входу (наприклад, `http://localhost:8080/api/v1`).

**Інтерцептори (Interceptors):** Інтегровано перехоплювачі запитів, які автоматично підставляють токен авторизації (`accessToken`) у заголовки `Authorization: Bearer ...` для захищених ендпоінтів.

**Обробка помилок:** Централізовано перехоплюються HTTP-помилки сервера (на кшталт `401 Unauthorized` або `409 Conflict`), що дозволяє коректно виводити інформативні повідомлення користувачу в інтерфейсі (через реактивні змінні `apiError`).

## 6. Ключові UI-компоненти та архітектурні рішення

### Адаптивна верстка та мобільна поведінка

Усі компоненти платформи (включаючи складні форми, такі як `AddProductForm.vue` та `CreateStoreForm.vue`) спроєктовані за допомогою сіток CSS Flexbox та Grid із використанням адаптивних префіксів Tailwind (`sm:`, `md:`, `lg:`). На мобільних пристроях блоки навігації, елементи форм та кнопки дій динамічно перебудовуються у вертикальні структури (`flex-col`), запобігаючи виходу контенту за межі екрана (Viewport).

### Робота з реактивними таймерами

У компоненті `CabinetGroupBuys.vue` реалізовано механізм відліку часу для групових покупок у реальному часі. Використовуючи бібліотеку `dayjs` (UTC plugin), фронтенд щосекунди вираховує різницю між поточним часом розробника та часом експірації сесії з сервера. При досягненні нуля інтерфейс реактивно оновлює статус картки на `EXPIRED` («Час вичерпано») без необхідності оновлення сторінки користувачем.
